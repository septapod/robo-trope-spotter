# Three-model eval: Opus 4.7 vs Sonnet 4.6 vs Haiku 4.5

Run on 2026-04-30 against 49 hand-labeled passages.

## Methodology

Each passage was sent to each model in single-pass detection mode using the production system prompt verbatim. No Haiku validation pass, no em-dash regex injection — those are pipeline artifacts that would muddy a model-vs-model comparison.

Detections were scored by tropeId match against hand-labeled ground truth. Severity tier comes from the canonical registry, not the model output. Recall and precision are computed across all detections; per-tier metrics group by trope tier.

Test set composition:
- linkedin-post: 16
- vendor-white-paper: 8
- human-blog: 7
- consultant-email: 6
- newsletter: 4
- edge-case: 6
- human-edited-ai: 2

## Recommendation

### TL;DR

Keep Sonnet 4.6 as the primary detector. Use Haiku 4.5 as the degraded-mode tier in the Energy Meter cascade. Drop to a regex-only floor below that, not Haiku-only — Haiku's precision drops too far on long-tail patterns to make the second tier meaningful as a fallback. Opus 4.7 is meaningfully better than Sonnet, but the gain (10pp recall, 9pp precision) does not justify 2x cost for a free public tool, with one caveat below.

### Per-tier behavior

| Tier | Opus | Sonnet | Haiku | What this means for the cascade |
| --- | --- | --- | --- | --- |
| 1 (Dead Giveaway) | 93% | 80% | 73% | All three pass the floor for catching the most-shareworthy tells. |
| 2 (Red Flag) | 89% | 78% | 59% | Sonnet still strong. Haiku noticeably weaker. |
| 3 (Worth Noting) | 82% | 72% | 65% | Opus pulls ahead. Haiku surprisingly close to Sonnet here. |
| 4 (Subtle Tell) | 80% | 80% | 20% | Sonnet and Opus tie. Haiku falls off a cliff. |
| 5 (Deep Cut) | 0% | 0% | 0% | **Test-set gap, not a model gap.** No Tier 5 ground truth labels in the current test set. See follow-on. |

### Cascade tier assignments

- **Caffeine (full power):** Sonnet 4.6 detection + Haiku 4.5 validation pass (current production pipeline).
- **Tea (slightly degraded):** Sonnet 4.6 detection only, no Haiku validation. Saves the validation cost (~30% of the per-analysis spend) and runs faster. Quality trade is real but bounded since precision was already in the 47% range with the validation pass active.
- **Water (pattern-matching only):** Regex-only heuristic engine, no LLM. The current `src/lib/analysis/heuristic-engine.ts` already implements this. Honest about what it catches and what it misses.
- **Napping:** Cap exhausted; come back tomorrow.

Haiku-only is **not** in the recommended cascade. Reasoning: Haiku's 41% precision (worse than Sonnet by 5pp) combined with 23 expected-non-trope violations (vs 18 for Sonnet) means switching to Haiku as a degraded tier would surface more false positives than the current pipeline. Users degraded to Haiku would see more aggressive flagging on clean human writing, which is the worst possible UX for a degraded mode. The regex-only floor is a better honesty signal: "we're rate-limited, here's what we can still catch without a smart model."

### When to consider Opus 4.7

The 10pp recall gap is real and concentrated where it matters most (Tier 1, 93% vs 80%). For the launch moment specifically, an Opus pass on a small subset of analyses (e.g., the first 50 published reports per day, or one Opus pass per IP per day) would noticeably reduce false negatives on the highest-shareworthiness tier. Cost overhead at 50 Opus passes/day: ~$2/day or $60/month. Worth considering as an upgrade tier, not as the default.

### Known caveats

1. **Tier 5 untested.** The current test set has no Tier 5 ground truth labels. The 0% recall reading is artifact, not signal. Follow-on: add 5-8 passages with hand-labeled Tier 5 patterns (low-burstiness, perfect-grammar, style-consistency) and rerun.
2. **Eval used single-pass detection.** Production runs Sonnet detection + Haiku validation. The validation pass actively rejects false positives, so production precision is materially higher than this eval's 47% reading for Sonnet. The eval measures *raw model behavior*, not the production cascade. A follow-on eval comparing production-shape pipelines (Sonnet+Haiku vs Sonnet-only vs Haiku+Haiku) would inform tier-2 design.
3. **Prompt has a real DISCUSSION-vs-USAGE bug.** All three models flagged at least one trope in Mollick's post (which discusses the patterns rather than using them). The prompt's DISCUSSION-vs-USAGE rule isn't being honored. This is a prompt-engineering issue, not a model-selection one. Worth a separate prompt-iteration sprint after launch.
4. **Some draft labels in `test-set.json` may be wrong.** The 49-entry test set was scaffolded by the agent with hand-crafted passages. Brent should spot-check failed cases (top-misses tables above) and correct any labels that are actually right by the model and wrong in ground truth. Re-run with `npm run eval -- --skip-run` after edits.

### Decision for U2

Cascade tier definitions for the Energy Meter:

- **Caffeine** = current pipeline (Sonnet detect + Haiku validate)
- **Tea** = Sonnet-only (skip the validation pass)
- **Water** = regex/heuristic only
- **Napping** = locked

This is enough specificity for U2 (Energy Meter cascade) to start.

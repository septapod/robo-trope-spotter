# Drizzle migrations

Authoritative SQL migrations for the production database.

## Baseline status

- `0000_silky_harrier.sql` is the baseline snapshot generated by
  `drizzle-kit introspect` against production on 2026-04-17. It captures
  `reports` and `share_events` as they existed at that moment. The SQL body is
  wrapped in a `/* ... */` block so `drizzle-kit migrate` records the migration
  as applied without trying to re-create tables that already live in prod.
- `meta/_journal.json` and `meta/0000_snapshot.json` are the tracked snapshots
  drizzle-kit uses to diff future schema changes against the baseline.
- The pre-bootstrap hand-written `drizzle/0000_share_events.sql` (which was
  applied to prod via `scripts/apply-share-events-migration.mjs`) is no longer
  in the tree. Its content is included in the baseline above.

## Adding a new migration

1. Edit `src/db/schema.ts`.
2. Run `npm run db:generate` to emit a new numbered `.sql` file here.
3. Review the SQL, commit it, and deploy. Vercel's build pipeline runs
   `npm run db:migrate` against prod.

Never run `npm run db:push` against production. It bypasses the tracked
migration history.

## First deploy after this change

`drizzle-kit migrate` will:

1. Create `drizzle.__drizzle_migrations` if missing.
2. Record `0000_silky_harrier` as applied (empty execution because the SQL body
   is commented out).
3. Apply any subsequent `000N_*.sql` files on future deploys.
